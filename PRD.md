# InternPilot 产品需求文档

## 1. 产品概述

**产品名称：** InternPilot  
**产品类型：** 实习申请管理 Web App  
**目标版本：** MVP  
**核心用户：** 正在同时申请多个公司、岗位、地区和截止日期的学生。

InternPilot 帮助学生把实习申请流程集中管理在一个清晰的工作台里。用户不再需要把岗位链接、申请截止日期、面试记录、跟进事项和 offer 决策分散在表格、备忘录、邮箱和浏览器书签中，而是可以在一个产品里追踪每一个机会从意向岗位到最终结果的完整状态。

## 2. 问题背景

学生在一个申请季中通常会同时申请大量实习岗位。随着申请数量增加，管理难度会迅速上升，因为每个岗位都有不同的截止日期、申请阶段、跟进动作、面试轮次和相关材料。

常见痛点：

- 容易忘记申请截止日期或后续跟进事项。
- 申请记录依赖手动表格，维护成本高且容易混乱。
- 面试记录、岗位信息和联系人信息分散在多个工具中。
- 用户很难一眼判断当前申请进度是否健康。
- 用户不知道今天最应该优先处理哪一个申请。

## 3. 产品目标

MVP 需要帮助用户完成以下目标：

- 用结构化方式记录实习申请。
- 通过可视化流程看清每个申请所处阶段。
- 快速识别即将到期的申请和下一步行动。
- 快速搜索和筛选申请记录。
- 导出申请数据，便于备份或二次整理。

## 4. 非目标

MVP 阶段暂不包含：

- 用户账号和云端同步。
- 多人协作。
- 自动抓取招聘网站岗位。
- 邮箱或日历集成。
- 简历或 cover letter 生成。
- AI 推荐或自动分析功能。

以上能力可以作为后续版本迭代方向。

## 5. 目标用户

### 主要用户：正在集中申请实习的学生

**用户画像：** 大学生或研究生，在一个申请周期中申请 10-50 个实习岗位。  
**核心需求：** 可靠地管理申请记录、截止日期、面试进度和跟进动作。  
**当前替代方案：** Excel / Google Sheets、Notion 表格、手机提醒、浏览器书签和邮箱搜索。  
**成功时刻：** 用户打开 InternPilot 后，可以立刻知道今天最需要处理哪些申请。

### 次要用户：转专业或早期职业候选人

**用户画像：** 正在申请实习、学徒项目、graduate scheme 或初级岗位的候选人。  
**核心需求：** 清晰的申请流程视图和轻量级管理方式。  
**成功时刻：** 用户可以比较不同机会，并避免错过关键步骤。

## 6. 用户故事

- 作为用户，我希望添加新的实习申请，以便从一开始就追踪它。
- 作为用户，我希望编辑已有申请，以便更新状态、备注和下一步行动。
- 作为用户，我希望按申请阶段查看所有机会，以便理解当前 pipeline。
- 作为用户，我希望按公司、岗位或地点搜索，以便快速找到某个申请。
- 作为用户，我希望按状态或优先级筛选，以便聚焦当前最相关的申请。
- 作为用户，我希望看到即将到期的申请，以便不漏掉截止日期或 offer 决策。
- 作为用户，我希望导出申请数据，以便备份或在其他工具中继续使用。

## 7. MVP 范围

### 7.1 仪表盘

仪表盘需要展示：

- 已追踪申请总数。
- 活跃申请数量。
- 处于面试阶段的申请数量。
- 未来 7 天内到期的申请数量。
- 按紧急程度排序的关注队列。
- Pipeline 健康度指标。

### 7.2 申请管理

用户可以：

- 添加申请。
- 编辑申请。
- 删除申请。
- 记录公司、岗位、状态、优先级、截止日期、地点、下一步行动和备注。

必填字段：

- 公司
- 岗位

选填字段：

- 状态
- 优先级
- 截止日期
- 地点
- 下一步行动
- 备注

### 7.3 Pipeline 看板

申请需要按照以下阶段分组：

- Wishlist：意向岗位
- Applied：已申请
- Interview：面试中
- Offer：已获得 offer
- Rejected：已拒绝

每张卡片需要展示：

- 公司
- 岗位
- 优先级
- 截止日期状态
- 编辑入口

### 7.4 申请列表

申请列表需要展示：

- 公司
- 岗位
- 状态
- 截止日期
- 优先级
- 编辑入口

### 7.5 搜索与筛选

用户可以：

- 按公司、岗位或地点搜索。
- 按状态筛选。
- 按优先级筛选。
- 一键清空筛选条件。

### 7.6 数据持久化

MVP 阶段使用浏览器 `localStorage` 在本地保存申请数据。

用户刷新页面后，数据仍应保留。

### 7.7 数据导出

用户可以将所有申请导出为 CSV 文件。

CSV 需要包含：

- 公司
- 岗位
- 状态
- 优先级
- 截止日期
- 地点
- 下一步行动
- 备注

## 9. 非功能需求

- 应用不依赖后端即可运行。
- 应用在桌面端和移动端浏览器中都应快速加载。
- 界面需要响应式适配。
- 小屏幕下文字仍需清晰可读。
- 核心流程应尽量减少点击次数。
- MVP 阶段用户数据不应离开本地浏览器。
- 视觉风格应像一个严肃的效率产品，而不是营销落地页。

## 10. 用户体验要求

产品整体感受应当是：

- 清晰
- 克制
- 聚焦
- 有组织感
- 打开后立刻有用

关键体验原则：

- 仪表盘回答：“我现在最需要关注什么？”
- Pipeline 回答：“每个机会目前处于什么阶段？”
- 申请列表回答：“我要找的具体申请在哪里？”
- 添加和编辑流程要足够轻量，适合每天反复使用。

## 11. 信息架构

主要页面：

- **Overview：** 指标、关注队列、Pipeline 健康度。
- **Pipeline：** 按阶段分组的申请看板。
- **Applications：** 可搜索、可筛选的申请表格。

主要操作：

- 添加申请
- 编辑申请
- 删除申请
- 导出 CSV
- 重置演示数据

## 12. 数据模型

每条申请记录包含：

```json
{
  "id": "string",
  "company": "string",
  "role": "string",
  "status": "Wishlist | Applied | Interview | Offer | Rejected",
  "priority": "High | Medium | Low",
  "deadline": "YYYY-MM-DD",
  "location": "string",
  "nextAction": "string",
  "notes": "string"
}
```

## 13. 成功指标

MVP 验证指标：

- 用户可以在 30 秒内添加一条申请。
- 用户打开应用后 10 秒内可以识别最紧急的申请。
- 用户可以在 10 秒内通过搜索或筛选找到某条申请。
- 用户可以成功导出申请数据。
- 页面刷新后数据不会丢失。

如果未来做成线上产品，可进一步追踪：

- 周活跃用户数。
- 每个用户平均追踪的申请数量。
- 添加超过 5 条申请的用户占比。
- 一个申请周期内的留存率。
- 导出或提醒功能使用率。

## 14. 边界情况

- 当前没有任何申请。
- 搜索结果为空。
- 某条申请没有截止日期。
- 某条申请已经逾期。
- 用户编辑后刷新页面。
- 用户删除某个阶段中的最后一条申请。
- 用户筛选后 Pipeline 为空。

## 15. 风险与取舍

| 风险 | 影响 | 应对方式 |
| --- | --- | --- |
| 本地数据可能在浏览器缓存被清理后丢失。 | 中 | MVP 提供 CSV 导出；后续考虑云同步。 |
| 用户可能希望设置日历提醒。 | 中 | 在后续版本加入提醒功能。 |
| 固定阶段无法覆盖所有申请流程。 | 低 | 后续支持自定义阶段。 |
| 手动录入可能有重复劳动。 | 中 | 后续加入岗位链接解析或浏览器插件。 |

## 16. 后续路线图

### V1

- 看板阶段间拖拽。
- 截止日期提醒。
- 日历导出。
- 自定义阶段。
- 申请回复率和面试转化率分析。

### V2

- 用户账号和云同步。
- 上传或关联简历、cover letter 和岗位描述。
- 从邮件中解析申请状态更新。
- 保存岗位链接。
- 申请模板。

### V3

- AI 辅助下一步行动建议。
- 面试准备工作区。
- Offer 对比工具。
- Career center 或导师分享模式。

## 17. 验收标准

MVP 完成标准：

- 用户可以添加、编辑、删除、搜索、筛选和导出申请。
- 申请数据在页面刷新后仍然保留。
- 仪表盘能够准确反映当前申请数据。
- Pipeline 看板能够按状态正确分组申请。
- 应用在桌面和移动端屏幕上都可用。
- README 能说明产品定位和本地运行方式。
